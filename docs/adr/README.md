# Architecture decision records

Use an ADR when a change affects an approved requirement, module/data ownership, consistency boundary, security model, technology baseline, release sequencing, or another locked architectural decision.

1. Copy `ADR_TEMPLATE.md` to `ADR-NNNN-short-title.md`.
2. Keep its status `Proposed` while discussing the conflict and trade-offs.
3. Implement only after explicit approval changes the status to `Accepted`.
4. Update affected baselines, plans, traceability, and diagrams with the accepted decision.

Do not use an ADR to disguise convenience-driven scope growth.
